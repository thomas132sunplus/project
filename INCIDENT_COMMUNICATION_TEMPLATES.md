# Incident Communication Templates

## 1. 內部技術通報

### 主旨

`[高優先] Edgewalker Firestore 資料暴露事件通報與處置進度`

### 內容

各位團隊成員：

我們確認平台先前存在 Firestore 存取控制設定缺失，導致未登入者可直接透過 Firestore REST API 讀取部分 collections，其中包含 users collection 的個人資料欄位。

### 已確認事實

- 受影響資料主要來自 `users` collection。
- 可讀取欄位包含姓名、email、手機、學校、年級、角色與 Firebase UID。
- 先前 production build 也公開了 source map，增加攻擊者理解前端資料結構的難度降低。
- 目前無足夠稽核日誌可確認是否已遭第三方實際存取。

### 已完成修補

- Firestore rules 已改為預設拒絕，並於 production 生效。
- 匿名讀取 `users`、`teams`、`tournaments`、`activities` 已實測回 `403`。
- Vite production build 已停用 source map。
- Firebase Hosting 已補上 CSP 與其他安全標頭。
- Hosting rewrite 已調整，不存在的 `.map` 會回 `404`。
- 管理腳本已改為使用環境變數憑證路徑，不再依賴 repo 根目錄私鑰檔。
- Firebase Admin service account 金鑰已完成輪替，舊金鑰已刪除。
- Firebase Browser key 已完成 HTTP referrer 限制、刪除未使用舊 key，並縮到最小可用 API allowlist。
- Cloud Audit Logs 的 `管理員讀取`、`資料讀取`、`資料寫入` 已啟用。
- logs-based metric 與對應的敏感變更 alert policy 已建立。

### 尚待完成

- 補查 GitHub 遠端、PR、附件與外部分享位置是否曾包含舊私鑰。

### 作業要求

1. 停止使用任何舊的 service account JSON。
2. 不要將任何憑證檔案放入版本庫、聊天室或共享雲端資料夾。
3. 若任何人曾下載、轉寄或備份過舊金鑰，請立即回報。

## 2. 對外非技術版通報

### 主旨

`邊境之外平台個資保護事件說明`

### 內容

我們近日發現，平台先前有一項資料存取設定錯誤，可能導致未經授權的人在未登入情況下讀取部分使用者資料。

目前已知可能受影響的資料包含：姓名、email、手機、學校、年級、角色與系統帳號識別碼。

我們已經完成下列修補：

- 關閉未授權的資料庫公開讀取。
- 移除不應公開的除錯檔案。
- 補上網站安全保護設定。

截至目前，我們沒有足夠系統紀錄可以確認資料是否曾被第三方實際存取，但因涉及個人資料保護，我們以最高標準處理並完成封鎖。

我們已完成憑證輪替、金鑰限制與稽核強化，目前正進一步補齊告警通知通道與內部應變流程。

目前告警政策已綁定 Email 通知通道，已透過 API key 變更事件成功觸發 incident，且已確認收件信箱收到告警通知；後續僅剩外部通道殘留複查。

若你曾在平台上填寫個人資料，建議你：

1. 留意是否收到異常 email、訊息或來電。
2. 若你的手機或 email 也使用在其他重要服務，建議提高警覺。
3. 如有任何疑慮，可與我們聯繫。

對於造成的不安，我們致上歉意。

## 3. 家長 / 學校版通知

目前沒有足夠紀錄能確認資料是否被實際下載或利用，但我們選擇主動說明，以便家長與學校掌握情況。

如果需要，我們可以提供：

- 後續安全改善清單
